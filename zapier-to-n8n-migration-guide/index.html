The Zapier Bill That Broke The Camel's Back

Last quarter I audited a 34-person staffing firm paying $847/month to Zapier for 47 zaps. We rebuilt the same logic on n8n in nine days. Their new monthly cost: $31 in DigitalOcean droplet fees. That's a 96% reduction, and the new system does things their old one couldn't — branching logic, AI enrichment, retries that actually retry.

This guide walks you through the same migration, step by step, without breaking the automations your team relies on every morning at 8 AM.

Quick on who's talking:

  • AI & Automation Architect — 40+ systems deployed for service businesses
  • Built n8n + Claude workflows that have automated 6,000+ hours of repeat work
  • Work spans 8+ industries (construction, manufacturing, staffing, property management, agencies)
  • Focus on mid-market 11-100 employees, not enterprise, not solo founders

Why This Migration Matters Right Now

Zapier's per-task pricing was designed for a world where your business ran 500 automated tasks a month. Once you're past 10,000 tasks — which happens fast at 11-100 employees — every new customer makes your automation bill grow. That's backwards. Infrastructure should get cheaper per unit as you scale, not more expensive.

The second problem is structural. Zapier's Paths, Filters, and Formatter steps each consume tasks, so a single "lead routing" zap can burn 4-6 tasks per execution. I've watched a property management company hit their 50,000-task plan limit on the 19th of every month, then frantically upgrade mid-cycle at premium pricing. That's not a tooling problem. That's a business model problem.

Zapier charges you more as you succeed. n8n charges you the same whether you run 1,000 tasks or 1,000,000. At mid-market volume, that gap pays for an entire ops hire within 18 months.

1. Audit Your Current Zapier Setup

Before you migrate anything, you need a complete picture of what you're running. Most teams I work with underestimate their zap count by 30-40%. They remember the important ones and forget the cron-job zaps that quietly sync data at 3 AM.

Open your Zapier dashboard, filter by "On" status, and document every active zap. Use this template:

Identify Your Core Integrations

Look at the trigger and action columns. Which apps appear in 5+ zaps? Those are your core integrations. They need to work flawlessly in n8n before you migrate anything secondary. For most service businesses, the list looks like this:

  • CRM: HubSpot, Salesforce, Pipedrive, Close
  • Accounting: Xero, QuickBooks, Stripe
  • Communication: Slack, Gmail, Outlook, Teams
  • Project management: ClickUp, Notion, Asana, Monday
  • Forms & intake: Typeform, Tally, Jotform, Google Forms

Flag The Complex Ones

Roughly 70% of zaps migrate in under an hour. The other 30% need real engineering. Flag any zap that:

  • Uses Zapier-native features: Paths, Formatter, Filter, Sub-Zaps, Storage by Zapier
  • Has 5+ steps: these usually hide branching logic that needs careful translation
  • Connects to obscure apps: if the app isn't in n8n's 400+ node library, you'll need an HTTP Request node and the API docs
  • Touches money or contracts: invoicing, payments, MSA generation — test these in staging for at least a week before flipping over

The audit isn't optional. I've seen migrations stall for six weeks because someone forgot the zap that emails the bookkeeper every Friday. Write it all down before you touch n8n.

You have three deployment paths, and the right answer depends on your team's technical comfort.

For most service businesses in the 11-100 range, I recommend self-hosted on a $24/month Hetzner CX22 with PostgreSQL backing. It handles 50,000+ executions per month without breaking a sweat, and it gives you full control over data residency — which matters if you're in healthcare, legal, or financial services.

Production Hardening Checklist

  • Enable queue mode: separates main and worker processes so heavy workflows don't block the UI
  • Use external PostgreSQL: SQLite is fine for testing, terrible for production
  • Set up automated backups: daily database dumps to S3 or Backblaze B2
  • Configure SSL via Caddy or Nginx: never expose n8n on plain HTTP
  • Enable basic auth or SSO: n8n's built-in user management works fine for teams under 20

3. Migrate Your Credentials First

This is the step everyone underestimates. In Zapier, every zap stores its own connection. In n8n, credentials are reusable across workflows — which is better, but means you need to set them all up once before migrating any zaps.

Pull up your audit table from Section 1 and create n8n credentials for every unique app. For OAuth services like HubSpot or Google Workspace, you'll need to create your own OAuth app in their developer console. This adds 15-30 minutes per integration but eliminates Zapier's rate limiting and gives you a dedicated quota.

Credentials Setup Order

  1. Communication tools first (Slack, Gmail) — you'll use these for migration testing notifications
  2. Core data sources next (CRM, database, forms) — these power the most workflows
  3. Action services last (accounting, document generation, payment) — highest blast radius if misconfigured

Test every credential with a simple "fetch one record" workflow before you build anything real. Catching a bad OAuth scope now saves you debugging a broken production workflow at 11 PM on a Tuesday.

Translating Zapier Concepts To n8n

Most one-to-one translations are obvious. The tricky ones are below:

  • Zapier Paths → n8n Switch or IF nodes: Switch handles 4+ branches cleanly, IF for binary splits
  • Zapier Formatter → n8n Set, Code, or Date & Time nodes: the Set node handles 90% of cases without writing code
  • Zapier Filter → n8n IF node with the "false" branch left empty: the workflow simply stops on that path
  • Zapier Sub-Zaps → n8n Execute Workflow node: better than Zapier's version because you can pass complex objects
  • Zapier Storage → Supabase, Postgres, or n8n's static data API: for anything more than 10MB, use Supabase
  • Zapier Webhooks → n8n Webhook trigger: identical concept, but n8n gives you the full URL with no plan-based throttling

Run Both Systems In Parallel

For high-criticality workflows, run the n8n version alongside the Zapier version for 7-14 days. Send the n8n output to a Slack channel called #migration-shadow instead of taking real action. Compare the outputs every morning. When you've gone five business days with zero discrepancies, kill the Zap.

5. Handle The Edge Cases Zapier Hides From You

Zapier silently handles a lot of things you'll need to think about explicitly in n8n. This is good — it means you'll actually understand your workflows — but it's a learning curve.

Error Handling

In Zapier, a failed task shows up in your task history and maybe sends an email. In n8n, you build error handling into the workflow itself. Use the Error Trigger node to create a single error-handling workflow that catches failures from any other workflow. Route those errors to Slack with workflow name, node name, error message, and execution URL. You'll catch problems hours or days before your team would have noticed.

Rate Limiting

Zapier throttles you transparently. In n8n, when you hammer an API, you get the API's actual rate limit error. Use the Wait node, the Loop Over Items node with batch size, or the built-in retry logic on HTTP Request nodes. For HubSpot, that means batches of 100 with 200ms delays. For Notion, batches of 3 with 350ms delays.

Data Type Strictness

Zapier coerces types aggressively. n8n respects them. If a webhook sends a number as a string, n8n keeps it as a string until you explicitly convert it. Use the Edit Fields (Set) node with the type-conversion toggles, or a small Code node with Number(item.json.amount).

The first month on n8n, you'll find three or four zaps that have been silently dropping or corrupting data for years. Zapier hid the failures. n8n surfaces them. Treat this as a feature, not a bug.

7. The 30-Day Migration Plan

If you're staring at 40+ zaps and wondering where to start, here's the calendar I give clients.

Week 1: Foundation

  • Days 1-2: Complete the audit table from Section 1
  • Days 3-4: Stand up the n8n instance and harden it
  • Day 5: Set up all credentials and test each one with a hello-world workflow

Week 2: Easy Wins

  • Days 6-10: Migrate every workflow scoring 1-3 on the priority rubric
  • Goal: 50-60% of zap count migrated, 20-30% of cost saved

Week 3: The Hard Stuff

  • Days 11-15: Tackle Paths, multi-step workflows, and anything with custom Code by Zapier steps
  • Build the Error Trigger workflow and connect it to every migrated workflow

Week 4: Parallel Run + Cutover

  • Days 16-22: Run high-criticality workflows in shadow mode
  • Days 23-27: Flip workflows over one at a time, starting Monday morning so you have the full week to catch issues
  • Days 28-30: Downgrade or cancel Zapier, document the new system, train one backup person

The teams that fail at this migration try to do it in a weekend. The teams that succeed treat it like a real project with milestones, owners, and a rollback plan. Budget the calendar time, not just the labor hours.
