What To Do With This

Spend 1-2 hours mapping your current CRM-to-project-management flow against this architecture. If you already have automation in place, audit it against the two-stage model. If you're building from scratch, use this as your blueprint before you write a single trigger.

The goal: make sure data is complete before folders and tasks get created. Because fixing a broken automation after it's already spawned 47 incomplete project folders is way harder than building it right the first time.

Why This Matters

Here's what I see constantly: someone builds an automation that triggers when a deal hits "closed won" in HubSpot (or whatever CRM you're using). The automation fires, creates a client folder in ClickUp or Asana or Monday, maybe pulls in some templates, assigns some tasks. Looks great on paper.

Then reality hits.

The deal was marked closed won, but the account manager field was empty. The service line items weren't finalized. The primary company name had a typo. Now you've got a project folder that's missing critical context, wrong assignments, and templates that don't match what the client actually bought.

The fix isn't to slow down your team or add more manual steps. The fix is to build your automation in two stages - where stage one creates a minimal record that humans can verify, and stage two only fires when that record is confirmed ready.

This is what I call the two-stage rocket ship approach. Stage one gets you off the ground. Stage two gets you into orbit. You don't fire stage two until stage one is complete.

The Two-Stage Architecture

Stage 1: Create the Engagement Record

Trigger: CRM deal reaches a specific status (like "closed won payment received")

What happens:
  • Creates a minimal record in your project management system
  • This is NOT the full project folder with all the templates
  • This is a holding record - think of it as a staging area
What gets created:
  • A single record in your client engagement dashboard (or equivalent)
  • Basic deal info pulled from CRM: client name, deal value, service lines
  • Status set to something like "pending setup" or "awaiting verification"
What does NOT happen yet:
  • No project folders created
  • No templates duplicated
  • No task assignments
  • No team notifications
Why this matters: Your data in the CRM might be incomplete at the moment the deal status changes. Maybe the sales rep was rushing. Maybe a field wasn't required. Maybe the primary company association got skipped. Stage 1 creates a checkpoint where someone can verify everything looks right before the big automation runs.

Stage 2: Build the Project Infrastructure

Trigger: Human changes the engagement record status to "onboarding" (or your equivalent)

What happens:
  • Creates the full client folder in your delivery space
  • Populates all subfolders based on service lines
  • Pulls in the correct templates from your process library
  • Assigns account managers and team members
  • Sets up views, docs, and dashboards
Why the human trigger matters: By requiring someone to manually change the status to "onboarding," you're building in a verification step. The person changing that status is confirming: yes, the deal info is complete, yes, the service lines are correct, yes, we're ready to spin up the full project.

This isn't adding friction for the sake of it. This is making sure your automation builds the right thing.

Real Example: HubSpot to ClickUp

Here's how this works in practice with a HubSpot-to-ClickUp setup I built recently:

Stage 1 Trigger: Deal status changes to "closed won payment received" in HubSpot

Stage 1 Actions:
  • Automation creates a new record in the Client Engagement Dashboard (a database in ClickUp)
  • Pulls in: deal name, primary company, service line items, account manager
  • Sets record status to "active" or "pending"
Stage 2 Trigger: Someone manually changes that record's status to "onboarding" Stage 2 Actions:
  • Creates a new folder in the Delivery space with the client name
  • Inside that folder: creates subfolders for project management, client onboarding, offboarding, and documents
  • Pulls in templates based on what service lines were on the deal
  • If the deal had "HubSpot Implementation" as a service line, it pulls the HubSpot Implementation checklist from the process library
  • If the deal had "Website Build," it pulls the website build templates
  • Assigns the account strategist based on the HubSpot field
The key: all that template-pulling and folder-building only happens AFTER a human has looked at the engagement record and confirmed it's ready. No more half-built projects because the deal was closed before all the fields were filled.

Handling Existing vs. New Clients

One thing that breaks automations fast: creating duplicate folders for clients you already have.

The problem: Client "UTK" already has a folder. A new deal comes in for UTK. Your automation doesn't know the folder exists, so it creates a brand new one. Now you've got two UTK folders and everyone's confused.

The solution: fuzzy matching

Before creating a new folder, the automation should:
1. Pull a list of all existing client folders
2. Compare the incoming deal's primary company name against that list
3. If there's a 90%+ match, route to the existing folder
4. If no match, create a new folder

This handles variations like "UTK" vs "University of Tennessee" vs "UTK - Main Campus." The automation scores the similarity and makes a decision.

Pro tip: If your CRM has a "deal type" field with options like "new business" vs "existing business," you can use that as an additional signal. Existing business deals should almost always route to an existing folder. New business deals might need a new folder created.

But tbh, the fuzzy matching is more reliable than trusting your sales team to always pick the right deal type. Build the fallback.

Client Folder Organization Rules

Here's where things get nuanced. Not every client needs the same folder structure.

Single-Folder Clients

Some clients have multiple engagements over time, but all their work should go into one folder.

Example: A university client where every project - website redesign, academic calendar, foundation work - all goes into the same UTK folder.

How to handle it:
  • Build a rule that checks if the primary company or deal name contains certain keywords
  • If deal name contains "UTK" OR primary company is "universityofutk.edu," route to the existing UTK folder
  • Don't create new subfolders for each engagement

White-Label / Parent-Child Clients

Some clients have sub-clients. Agency setups, white-label relationships, that kind of thing.

Example: An agency called Evenbound has some clients who pay the agency directly (white-label), and some who pay you directly but were referred by the agency.

How to handle it:
  • White-label clients (where Evenbound is paying you) go into a root Evenbound folder
  • Direct-pay clients (where the end client is paying you) get their own separate folders
  • Trigger rule: if primary company = "Evenbound" OR deal name contains "Evenbound," route to Evenbound root folder
  • Otherwise, create a new client folder
The key: This logic has to be built into your automation rules. And your sales/BD team needs to know to set the primary company field correctly. Fields are more reliable than text matching on deal names, but having both as a fallback doesn't hurt.

What Needs to Be in the CRM BEFORE the Trigger

Your automation is only as good as your data. Here's what needs to be filled out in your CRM before that "closed won" status change:

    • Field: Primary Company | Why It Matters: Determines folder routing and fuzzy matching
    • Field: Deal Name | Why It Matters: Backup for matching, plus creates readable folder names
    • Field: Service Line Items | Why It Matters: Determines which templates get pulled
    • Field: Account Manager / Strategist | Why It Matters: Determines task assignments
    • Field: Deal Type (new vs existing) | Why It Matters: Additional routing logic
    • Field: Deal Value | Why It Matters: Useful for prioritization views
Critical rule: Information must be in the CRM before the deal triggers the automation. If someone marks a deal closed won and then goes back to add the account manager, the automation already fired. It's too late.

Train your team: fill the fields before changing the status. Or build validation rules in your CRM that prevent status changes when required fields are empty.

When Things Go Wrong (and How to Fix Them)

Even with two-stage architecture, stuff breaks sometimes. Here's the reality:

Problem: Wrong folder created because of a typo in the company name
Fix: It's not hard to relocate things in ClickUp. If a new folder was created with lists inside it, you just drag the lists into the correct folder. A few minutes of cleanup, not a disaster.

Problem: Template didn't pull because service line was missing
Fix: Go back to your process library, find the template, duplicate it into the correct folder. Then figure out why the service line was missing and fix your CRM validation.

Problem: Account manager wasn't assigned
Fix: The field in HubSpot was probably empty when the deal triggered. Manually update the assignment, then update the HubSpot record so it's right for future reference.

The point: Two-stage architecture reduces errors significantly, but it won't catch everything. Build your system to minimize errors, not to achieve perfection. Perfection doesn't exist, and chasing it will slow you down.

Moving Docs and Views to Folder Level

Quick operational improvement that makes a big difference: put your notes and docs at the folder level, not buried inside lists.

When you open a client folder, you want to see the key documents right there. Not three clicks deep inside a list inside a subfolder.

What this looks like:
  • Client folder opens
  • Notes and docs visible immediately in the folder view
  • GPT context docs, onboarding checklists, SOPs all accessible without hunting
Limitation to know: Some project management tools (ClickUp included) have API limitations around automated view creation. You might need to manually set up certain views even if the folders and docs are auto-populated. Check your tool's API docs before promising full automation.

Putting It All Together

The two-stage architecture solves three problems at once:

1. Data completeness - Stage 2 only fires when a human confirms the record is ready
2. Folder routing - Fuzzy matching and field-based rules put projects in the right place
3. Template accuracy - Service line items determine exactly which templates get pulled

It's not complicated, but it requires discipline. Your CRM data has to be clean. Your team has to know the system. And someone has to actually change that status to "onboarding" instead of assuming the automation handles everything.

The automation handles the grunt work. Humans handle the judgment calls. That's the split.

Your Next 7 Days

Day 1-2: Map your current CRM-to-project-management flow. Document every trigger, every action, every field that gets pulled. Identify where data completeness breaks down.

Day 3-4: Design your two-stage architecture. Define what Stage 1 creates (minimal engagement record) and what Stage 2 creates (full project infrastructure). Identify the human verification step between them.

Day 5: Build your fuzzy matching logic. Pull a list of existing client folders. Define your similarity threshold (90% is usually right). Test with variations of client names.

Day 6: Define your folder routing rules. Which clients get single folders? Which have parent-child relationships? Document the field-based and text-based triggers for each scenario.

Day 7: Train your team. Make sure everyone knows: fields get filled before status changes. The automation only works if the data is there first.
