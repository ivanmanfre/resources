Why This Migration Matters Right Now

Zapier is perfect for the first 18 months of a business. Point, click, connect two apps, done. But once you're an 11–100 person service company running real operations — dozens of zaps, multi-step flows, transformations, approvals — the pricing model and the platform limits start working against you.

The per-task billing scales linearly with your success. Every new customer, every new ticket, every extra step in a workflow increases your bill. I've audited Zapier accounts at $640/month running logic that, rebuilt on n8n, costs $35/month in server spend. The Zapier bill wasn't the worst part — it was the engineering debt piled up from 12-step zaps held together with Formatter, Filter, and Paths.

The question isn't "is n8n cheaper?" It's "is your current stack the one you'd build today if you were starting fresh at your current size?" For most 11–100 employee service businesses, the answer is no.

n8n flips the model. Self-hosted (or n8n Cloud if you'd rather skip ops) means you pay for infrastructure, not executions. The visual builder is more powerful than Zapier's, and when you need real code, you drop into a Code node and write JavaScript or Python. It's also the only sensible place to run AI agent workflows with Claude, OpenAI, or local models — try doing that in Zapier with 14 steps and you'll understand why I moved every client over.

1. Audit Your Current Zapier Setup

Before you migrate anything, you need to know what you actually have. Most teams are surprised — the 8 "main" zaps they remember are often 40+ in practice, half of them forgotten or duplicated.

Export Your Zap Inventory

Open your Zapier dashboard, filter to active zaps only, and build this table in a spreadsheet. Do not skip the last two columns — they drive every decision downstream.

Zap Name Trigger Action Apps Monthly Tasks Criticality Last Edited
New Lead → Slack + CRM HubSpot form Slack, Sheets ~900 High 3 months ago
Invoice Paid → Ops Notify Xero Gmail, Notion, Slack ~180 High 6 months ago
Typeform → Pipedrive Typeform Pipedrive, Gmail ~350 Medium 14 months ago
Daily Sheet Digest Schedule Sheets, Gmail ~30 Low 2 years ago

Identify Your Core Integrations

Look for the apps that show up in 60%+ of your zaps. These are load-bearing. They need to work flawlessly in n8n before you touch anything else. In the service businesses I work with, the core set is almost always:

  • CRM: HubSpot, Pipedrive, Salesforce, or Close
  • Accounting: Xero or QuickBooks
  • Comms: Slack, Gmail, Microsoft 365
  • PM / ops: ClickUp, Notion, Asana, Monday
  • Forms & intake: Typeform, Tally, Jotform
  • Storage: Google Drive, Dropbox, S3

Flag the Complex Ones

Some zaps migrate in 20 minutes. Others take half a day. Flag anything that hits one of these:

  • Zapier-native features: Paths, Formatter, Filter, Digest, Sub-Zaps, Tables
  • Step count: more than 5 actions
  • Obscure apps: anything without an obvious n8n node (check n8n.io/integrations before you panic — 400+ nodes exist)
  • Revenue-critical: invoicing, payment confirmations, onboarding, SLA alerts

Don't start migrating randomly. I've watched a 22-person agency break their client intake for 4 days because someone "just moved the easy ones first" and the easy ones were dependencies of the hard ones.

2. Score and Prioritize What to Migrate

Not every zap should move. Some should be retired entirely (you'd be surprised — 20-30% of zaps in most accounts haven't produced business value in 6+ months). Use this rubric on each zap:

Factor Score 1 Score 3 Score 5
Monthly task volume < 50 50–500 500+
Business criticality Nice to have Important Revenue-blocking
Migration complexity 1–2 steps, common apps 3–5 steps, some logic 6+ steps, Paths, or custom code
Cost savings if moved < $10/mo $10–50/mo $50+/mo

Add the scores. Anything 14+ migrates in wave one. 10–13 goes wave two. Below 10, ask whether it should exist at all before you spend time rebuilding it.

3. Stand Up n8n Before You Touch a Zap

The cardinal rule: get n8n running and stable before you migrate a single workflow. Trying to stand up infrastructure and rebuild critical automations in the same week is how you end up in a Slack war room on a Friday night.

Pick Your Hosting Model

Option Monthly Cost Best For Tradeoff
n8n Cloud (Pro) $50–$150 Teams that don't want ops Execution limits, less flexibility
Self-host on Hetzner / DigitalOcean $20–$60 Teams with any DevOps capacity You maintain uptime, backups, upgrades
Self-host on Railway / Render $20–$80 Middle ground, fast setup Less control than raw VPS, more than Cloud

For service businesses at 11–100 employees, I default to self-hosted on a $20/month Hetzner CX22 or DigitalOcean droplet with Docker Compose, managed Postgres, and Caddy for TLS. It's boring, it's stable, and it costs less than one Zapier seat.

Baseline Setup Checklist

  1. Deploy n8n in Docker with Postgres (not SQLite — SQLite is fine for demos, not production)
  2. Set up daily Postgres backups to S3 or Backblaze B2
  3. Enable SSL with a real domain (automations.yourcompany.com)
  4. Turn on user management and force 2FA for anyone with editor access
  5. Set the EXECUTIONS_DATA_PRUNE env vars so your database doesn't balloon
  6. Connect every core credential (CRM, email, Slack) and run a test workflow on each before you migrate anything real

4. Map Zapier Concepts to n8n Equivalents

Most migration pain comes from trying to rebuild Zapier 1:1. Don't. n8n's model is different — often better, sometimes just different — and forcing Zapier patterns onto it creates messy workflows.

Zapier n8n Equivalent Notes
Trigger Trigger node (Webhook, Schedule, App trigger) Webhook triggers are instant in n8n; Zapier polls on lower plans
Action Regular node Same concept, more parameters exposed
Paths Switch or IF node Switch handles 4+ branches cleanly
Filter IF node or Filter node IF stops the branch, Filter drops items from a list
Formatter Set, Edit Fields, or Code node Code node with JS is often faster than chaining 4 Formatter steps
Digest Schedule + Merge + aggregator More setup, infinitely more flexible
Sub-Zap Execute Workflow node Pass data in/out cleanly — build reusable sub-workflows
Storage / Tables Postgres, Supabase, or Airtable I use Supabase — cheap, vector-ready, proper SQL

The second you have more than three Formatter steps in a Zap, rebuild it as one n8n Code node. 15 lines of JavaScript replaces the entire chain, runs faster, and is actually readable in 6 months.

5. Execute the Migration in Waves

Never do a big-bang cutover. Run Zapier and n8n in parallel for 2–4 weeks while you build confidence. Here's the wave plan I give clients:

Wave 1: Low-Risk, High-Volume Zaps (Week 1)

Pick 3–5 zaps that are high volume but low blast radius. Think: notifications, logging, internal digests. These let you stress-test n8n on your infrastructure without betting the business.

  • Goal: prove n8n handles your peak volume
  • Success metric: zero failed executions over 5 business days
  • Rollback: re-enable the Zap — takes 10 seconds

Wave 2: Core Operational Workflows (Weeks 2–3)

Now move the revenue-adjacent work: lead routing, CRM syncs, task creation. These need careful parallel running. Keep the Zap active but mute its notifications, let n8n handle the real action, and compare logs daily.

  • Parallel run: both systems trigger, only n8n takes the business action
  • Daily diff: spreadsheet comparing Zapier run count vs n8n run count vs actual records created
  • Cutover criteria: 5 consecutive days with matching counts

Wave 3: Revenue-Critical and Complex (Week 4+)

Invoicing notifications, onboarding sequences, anything customer-facing. These get full parallel runs for at least 10 business days, with a named human checking outputs daily. No shortcuts.

Wave What Moves Parallel Run Owner
1 Internal notifications, logs, digests 3 days Automation lead
2 CRM, task creation, intake 5–7 days Ops manager
3 Billing, onboarding, customer comms 10+ days Founder / COO sign-off

6. Build in Error Handling Zapier Never Gave You

This is where n8n earns its keep. Zapier's error handling is primitive: the zap fails, you get an email, you go fix it. n8n lets you design real resilience in.

  • Error workflows: every production workflow should have a linked Error Workflow that catches failures, logs them to Supabase or a Slack channel, and — where safe — retries with backoff
  • Retry on fail: set at the node level; 3 retries with 5-second backoff handles 90% of transient API hiccups
  • Dead letter queue: for anything involving money or customer promises, write failed items to a Postgres table so nothing is ever silently dropped
  • Execution data retention: keep 30 days of logs; you'll thank yourself the first time a client asks "did our form submission go through on Tuesday?"

If a workflow touches revenue, it needs three things Zapier rarely enforces: idempotency, an error workflow, and a dead-letter table. Skip any of the three and you'll find out which one you needed the hard way.

7. Measure, Decommission, and Harden

Migration isn't done when the workflows run — it's done when Zapier is off and your team trusts n8n. Plan for a 30-day post-migration period before you shut down Zapier entirely.

Metrics That Actually Matter

  • Execution success rate: target 99.5%+ for core workflows
  • P95 execution time: watch for workflows that creep above 30 seconds — usually a sign you need to parallelize or add Wait/Queue nodes
  • Cost per 1,000 executions: should drop 60–90% vs Zapier; if it didn't, you over-provisioned your VPS
  • Recovery time: how long from a node failure to a human knowing? Under 5 minutes is the bar

Decommission Checklist

  1. Export every Zap's config as JSON (Zapier's export is limited but do it anyway)
  2. Screenshot each Zap for reference
  3. Pause — don't delete — Zaps for 30 days after their n8n equivalent is live
  4. Downgrade Zapier plan to Free tier at day 30 if all is well
  5. Cancel at day 60 once you've survived a full billing cycle on n8n without incident
