What To Do With This

Before you start your next document automation project, spend 30-45 minutes going through this checklist with whoever owns your IT infrastructure.

If you can't answer these questions confidently, you're not ready to start building. And that's fine. Better to discover the blockers now than three weeks into a project that's already over budget.

Why This Matters

Here's a scenario I see constantly: Your document automation project is 3 weeks behind and 40% over budget. The automation itself took 2 days. The other 3 weeks? Getting access to the damn files.

I worked with an insurance company recently. They needed to automate their policy document signing workflow. The actual automation, reading emails, triggering signature requests, saving signed documents back to folders, that part took maybe a week to build and test.

But getting the automation access to their network folders? That turned into a whole thing. Their documents lived on a cloud-based server that required VPN plus username/password authentication. We ended up needing to set up a remote machine on their server just so the automation could reach the files.

The automation wasn't the problem. The access layer was the problem. And nobody scoped for it upfront.

This checklist exists because I got tired of projects stalling on infrastructure issues that should have been identified in week one. Access layer problems typically add 2-4 weeks to project timelines when they're discovered mid-build. When you identify them upfront, you can plan for them, budget for them, or sometimes even eliminate them entirely.

The 3-Layer Checklist

Document automation projects have three layers that need to be scoped separately. Most teams only think about Layer 1 and wonder why everything takes three times longer than expected.

Layer 1: The Document Layer

This is the layer everyone thinks about. What documents are we automating?

Questions to Answer

1. What document types are in scope?
List every document type that will flow through this automation. Policy schedules, contracts, invoices, proposals, certificates, whatever applies.

2. What's the current format for each document type?
  • PDF generated from software (like Crystal Reports)
  • Word documents
  • Excel exports
  • Scanned images
  • Something else
3. What needs to happen to each document?
  • Digital signatures (how many signers per document?)
  • Data extraction
  • Template population
  • Format conversion
  • Routing/approval workflows
4. What's the volume? How many documents per day/week? This matters for tool selection and infrastructure planning.

5. Are there multiple signers?
If signatures are involved, does every person sign every page? One of my clients needed three different people to sign every page of every document. That's a very different automation than a single signer on a cover page.

6. What happens after the document is processed?
  • Saved back to same location?
  • Saved to different location?
  • Emailed to someone?
  • Archived?
  • All of the above?

Red Flags at This Layer

  • "The customer just sends us everything" (meaning you'll get 300-page PDFs when you only need 4 pages)
  • No clear list of document types
  • "It depends" answers to most questions
  • Signature requirements that aren't documented anywhere

Layer 2: The Access Layer

This is the layer that kills projects. Where do the documents live, and how will the automation get to them?

Questions to Answer

1. Where are the source documents stored?
  • Cloud storage (Google Drive, Dropbox, OneDrive, SharePoint)
  • Local network folders
  • Cloud-based server with authentication
  • Inside another software application
  • Email attachments
2. What authentication is required?
  • None (publicly accessible)
  • Username/password only
  • VPN required
  • Multi-factor authentication
  • Certificate-based authentication
3. Can the automation access the storage directly? This is the big one. If your files are on a network folder behind a VPN, your automation can't just reach in and grab them. You need a bridge. 4. What bridge options exist?
  • Remote machine on the server that can run automation
  • API access to the storage system
  • Export/sync to a location the automation CAN access
  • RPA (browser automation) using VPN credentials
5. Who controls access permissions? Get the name and contact info of the person who can actually grant access. Not their boss. The person with the keys.

6. What's the typical turnaround for access requests?
Some organizations can spin up credentials in a day. Others take 6 weeks and three committee meetings. Know which one you're dealing with.

Red Flags at This Layer

  • "You need to be in the office to access those folders"
  • VPN required but no plan for how automation will use VPN
  • IT team wasn't consulted before project kickoff
  • "We've never given system access to an automation before"
  • Nobody knows who can grant access permissions

The Remote Machine Option

When I hit VPN requirements, the cleanest solution is usually a remote machine hosted on the client's server. The automation runs on that machine, which already has network access. Then it communicates results out via API.

One client described it as: "What if you used a remote machine? We do have some remote machines on the server that you could load the application onto."

If that option exists, take it. It sidesteps most of the access headaches.

Layer 3: The Integration Layer

This layer is about how the automation connects to the tools that actually process the documents.

Questions to Answer

1. What tools are currently used for document processing?
  • Adobe Acrobat (for signatures)
  • DocuSign
  • Other e-signature platforms
  • Manual printing and scanning (yes, some companies still do this)
2. Do those tools have APIs? Adobe Acrobat Sign has an API. DocuSign has an API. Most modern tools do. But you need to verify, not assume. 3. What's the trigger for the automation?
  • Email arrives in specific folder
  • Document uploaded to specific location
  • Manual button press
  • Scheduled time
  • Status change in another system
4. What orchestration tool will coordinate everything? For most of my projects, I use n8n. It costs around $50-60/month and handles the orchestration between all the pieces. It also gives you a platform for future automations. 5. What notifications need to happen?
  • Who gets notified when documents need signing?
  • Who gets notified when signing is complete?
  • What happens when something fails?

Red Flags at This Layer

  • Current tools have no API and no plans to switch
  • Trigger mechanism is unclear ("someone will know when it needs to happen")
  • No budget for orchestration tools
  • Expected to integrate with legacy software that hasn't been updated in years

Putting It All Together

Before you greenlight a document automation project, you should be able to complete this summary:

Document Layer Summary
  • [ ] All document types listed with formats and volumes
  • [ ] Processing requirements clear for each type
  • [ ] Signature requirements documented (number of signers, pages per signer)
  • [ ] Destination for processed documents defined
Access Layer Summary
  • [ ] Storage location identified
  • [ ] Authentication requirements documented
  • [ ] Access path for automation confirmed (direct API, remote machine, or workaround)
  • [ ] Access request submitted or contact identified
  • [ ] Timeline for access estimated
Integration Layer Summary
  • [ ] Processing tools identified with API availability confirmed
  • [ ] Trigger mechanism defined
  • [ ] Orchestration platform selected
  • [ ] Notification requirements documented

Implementation Priority

If you can only do three things before starting a project, do these:

1. Confirm automation can access the files.
Not "probably can" or "IT said maybe." Actually confirm it. Test it. If there's a VPN requirement, figure out the workaround before you write a single line of automation.

2. Verify the signature tool has API access.
If you're automating document signing and the tool doesn't have an API, you're either using RPA (slower, more fragile) or switching tools. Know which one before you scope the project.

3. Get IT involved in week one, not week four.
The access layer issues that add 2-4 weeks to projects aren't technical problems. They're communication problems. The solutions usually exist. Someone just needs to ask the right person the right questions early enough.

Your Next Steps

Take this checklist into your next document automation planning session. If you get stuck on any layer, that's your signal to pause and solve that problem before proceeding.

The goal isn't to have perfect answers before you start. It's to know where the unknowns are so you can budget time and money for resolving them.
