Week two is where document automation projects die, and it's almost never the automation's fault.

Last year I watched a $14,000 signing workflow turn into a $31,000 project because nobody asked, on day one, how the automation was going to reach the actual PDFs. The build took four days. The VPN, the service account, the IP whitelist, and the "we'll need to loop in the MSP" email chain took six weeks.

If your last automation project slipped past deadline and budget, odds are strong the scope was written around the wrong layer.

Why This Matters

Here's the pattern I see at almost every 11-100 employee service business I scope: the document automation itself is a two-to-five day build. The access layer — the VPNs, the file permissions, the service accounts, the proprietary system that has no API — is a two-to-six week build. And it's almost never in the quote.

Insurance brokerages signing 50+ documents a day. Construction firms passing around 300-page bid packets. Property management companies routing lease addenda between four stakeholders. The work on top looks different. The thing that blows up the timeline is identical: nobody scoped infrastructure access before committing to a delivery date.

If you quote a document automation project before you've answered "how does the bot reach the file," you are not quoting a project. You are quoting a guess.

This guide gives you a three-layer checklist to run through before you sign a statement of work, pick a vendor, or assign an internal engineer. Spend 45 minutes on it with whoever owns infrastructure and whoever owns the workflow. It will save you a month.

1. The Three Layers, and Why Most Scopes Only Cover One

Every document automation project has three layers stacked on top of each other. The scope usually addresses the top layer in detail, waves at the middle, and completely ignores the bottom.

Layer What It Covers Typical % of Real Effort Typical % of Original Quote
Layer 1: Access VPN, permissions, service accounts, API keys, IP whitelisting, MFA 40-60% 5-15%
Layer 2: Extraction & Handling Reading PDFs, parsing fields, handling scans, dealing with proprietary formats 20-30% 25-35%
Layer 3: Automation Logic Triggers, routing, signing, approval chains, notifications 15-30% 50-70%

The gap between "real effort" and "original quote" on Layer 1 is where your project goes to die. The automation logic — the exciting part with the flowcharts and the Slack notifications — is usually the smallest chunk of real work. It just looks like the biggest chunk because it's the part everyone can visualize in the kickoff meeting.

Scope the access layer first. Scope it in writing. If the vendor won't commit to it before the build, the overrun is already baked in.

2. Layer 1 — Document Access (The Hidden Timeline Killer)

This is where 80% of document automation projects stall. Before you think about routing or signing or approval logic, answer this: can the automation physically reach the documents, on its own, without a human clicking something?

2.1 Where do the documents actually live?

  • On-premise file server: Usually means VPN or a jump box. Expect 1-3 weeks of IT coordination.
  • Cloud storage (Google Drive, SharePoint, Dropbox): Best case. Native APIs exist. Budget 2-5 days.
  • Proprietary software (AMS, ERP, practice management, job management): Check for an API before anything else. If there isn't one, you're in RPA or screen-scraping territory, and the cost doubles.
  • Email inboxes: Fine for intake, fragile for storage. Treat email as a trigger, not a system of record.
  • Multiple locations at once: The silent killer. Map every location before scoping.

2.2 What's required to get in?

  • Username + password only: Cleanest path. Service account recommended.
  • VPN + credentials: Workable, but the automation host now needs to sit inside the VPN or spin up a tunnel.
  • IP whitelisting: Your automation needs a static egress IP. Lambda and serverless won't work out of the box.
  • MFA / hardware token: Kills most cloud automation paths. You either provision a service account that bypasses MFA, or you run the automation on a dedicated machine inside the network.
  • Must be on a specific machine: You're building RPA, not automation. Budget accordingly.

2.3 Access path decision table

Access Method Works for Automation? Added Cost Notes
Direct API (REST / GraphQL) Yes — best option $0 Always check docs before scoping alternatives
VPN + service account Yes, with a host inside the network $500-$2,000 setup Need a VPS or on-prem runner
Remote desktop / RPA Yes, but fragile $2,000-$6,000 Breaks on every UI update
Manual export + watched folder Partial — not true automation $0 Keeps a human in the loop
MFA-required portal, no API Usually no N/A Push for vendor API access instead

2.4 Critical questions before scope is signed

  • Does IT permit service accounts? If not, every automation is tied to a person who will eventually leave.
  • Who provisions access? Internal IT, an MSP, or the software vendor themselves? Each adds a different lead time.
  • What's the SLA on access changes? Some MSPs treat "add firewall rule" as a two-week ticket.
  • Is there a dev/sandbox environment? If you're automating against production on day one, you're going to break production on day one.

3. Layer 2 — Document Extraction & Handling

Once the automation can reach the file, you have a second problem: can it actually read what's inside? This is where PDFs humble everyone.

3.1 Document type audit

  • Digitally generated PDFs: Text is selectable. Parsers work. Budget light.
  • Scanned PDFs: Images masquerading as documents. Require OCR. Expect 80-95% accuracy at best, which means a human review step.
  • Mixed PDFs: Digital text on pages 1-5, scanned signatures on page 6. The worst case. Build assumes you can detect which is which.
  • Proprietary formats (XFA forms, encrypted PDFs, DWG, IFC): Specialized tooling required. Validate support before quoting.
  • Office files (DOCX, XLSX): Usually easy, but watch for macros and embedded objects.

3.2 Volume and variance

A scope that handles 50 documents a day with three variations is cheap. A scope that handles 50 documents a day with 30+ template variations, including one-off custom forms, is not. Ask the workflow owner to pull the last 30 days of real documents and count templates. The number will surprise them.

Template variance, not document volume, is the cost driver. 5,000 copies of one template is easier than 500 copies of 200 templates.

3.3 Field-level scoring rubric

Factor Easy (1 pt) Moderate (3 pts) Hard (5 pts)
Document format Digital PDF / DOCX Mixed digital + scanned Pure scanned / handwritten
Template count 1-3 4-10 10+ or ad-hoc
Field extraction Named form fields Fixed layout positions Free-text inside paragraphs
Accuracy requirement 95%, human review OK 99%, sampled QA 99.9%, zero tolerance (legal, medical)

Score each document type. A total above 14 means you need a human-in-the-loop review step or you will regret shipping.

4. Layer 3 — Automation Logic & Routing

This is the part everyone wants to talk about at the kickoff meeting. It's also the most forgiving layer, because the tooling (n8n, Zapier, Power Automate, custom code) is mature and the patterns are well-known.

4.1 Map the workflow before the build

  1. Trigger: What event starts the automation? New file in folder, email arrival, form submission, scheduled sweep?
  2. Decision points: Who or what decides where the document goes? Is that decision rule-based, judgment-based, or mixed?
  3. Approval chain: How many humans sign? In what order? Can they approve in parallel?
  4. Exceptions: What happens when a signer rejects? When a document fails validation? When the external system is down?
  5. Audit trail: What's the evidence requirement? Who looks at logs, and when?

4.2 Where budget actually goes

  • Happy path: 20% of build effort.
  • Exception handling: 40% of build effort.
  • Notifications and audit: 15% of build effort.
  • Admin UI / reporting: 25% of build effort.

If a vendor's quote puts the happy path at 70% of the estimate, they are underpricing exception handling and you will pay for it in change orders.

5. Red Flags That Predict a Blown Timeline

Before you sign the SOW, run the project against this list. Each yes is worth one week of slippage, in my experience.

  • IT wasn't in the kickoff call: Layer 1 is about to become someone else's problem.
  • No sandbox environment: Every bug fix becomes a change management ticket.
  • "We'll figure out access during the build": You won't. You'll figure it out in week four when the build is done and the bot has nothing to read.
  • No named document owner: Scope will drift every time a new template surfaces.
  • Vendor hasn't asked about document volume or variance: They're quoting theory, not your reality.
  • MFA-locked portal with no API: You're buying RPA whether you know it or not.

Three or more yeses on this list and the project should either be re-scoped or paused. Shipping forward with that many unknowns is a negotiating position for a change order, not a plan.

6. A Realistic Budget Model

For a mid-market service business (11-100 employees) automating document intake, routing, and signing across one department, here's what realistic scoping looks like in USD.

Component Simple Project Typical Project Complex Project
Access layer (Layer 1) $1,500 - $3,000 $4,000 - $8,000 $10,000 - $20,000
Extraction layer (Layer 2) $2,000 - $4,000 $5,000 - $10,000 $12,000 - $25,000
Automation logic (Layer 3) $3,000 - $6,000 $7,000 - $14,000 $15,000 - $30,000
Total range $6,500 - $13,000 $16,000 - $32,000 $37,000 - $75,000
Typical delivery 3-5 weeks 8-14 weeks 16-26 weeks

If a quote comes in far under the simple range, the access layer has not been scoped. If it comes in far over the complex range without a clear explanation, the vendor is pricing in unknown-unknowns as margin.

7. How to Run This Checklist Internally

  1. Block 45 minutes with three people: workflow owner, infrastructure owner, and the person who will sign the invoice.
  2. Pull 30 days of real documents before the meeting. Count templates, identify scans, note exceptions.
  3. Walk the three layers in order: Access first, extraction second, logic third. Do not skip ahead, even when the logic is the fun part.
  4. Write down every "we'll figure it out later": Those are your real risks. Convert them into assumptions in the SOW or into pre-work tasks before the build starts.
  5. Share the output with any vendor you're quoting. Vendors who engage with it seriously are the ones worth hiring. Vendors who dismiss it are the ones who'll hit you with change orders in week four.
