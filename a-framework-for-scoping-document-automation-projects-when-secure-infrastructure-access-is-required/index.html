Why This Matters

Here's the pattern I see constantly across insurance firms signing 50+ documents a day, construction outfits dealing with 300-page PDFs nobody can open, and professional services teams drowning in approval workflows. The automation work — triggers, signing flow, notifications, routing — typically takes 2 to 5 build days. The surrounding project drags on for 4 to 7 weeks. The gap is almost entirely the access layer.

Teams quote based on the visible automation work, then discover they need VPN credentials, remote desktop provisioning, API tokens the vendor won't release, or IT tickets that sit in a queue for 11 business days. That discovery phase alone adds 2 to 4 weeks to a typical project. Sometimes it doubles the original timeline. I've watched a $12,000 project balloon into a $28,000 project because nobody asked where the PDFs actually lived until the day the build was supposed to start.

The automation is not the project. The access path to the documents is the project. Everything else is 15% of the work.

This guide is the scoping framework I use before quoting any document automation work. Run it before you sign a statement of work, before you commit a timeline, and before you tell your team it'll be done by end of quarter.

1. Map Where Documents Actually Live (Not Where You Think They Live)

The first scoping failure is assuming documents live in one place. In 40+ deployments, I have never once found a service business where documents lived in a single system. There's always a "but also" — contracts in SharePoint but signed copies in a shared drive, invoices in QuickBooks but approvals in email, job files in a proprietary construction platform but submittals on a field tablet.

Before anything else, build a document inventory. Walk the actual workflow and list every location a document touches from creation to archive.

2. Classify The Access Path For Each Document Stream

Once you know where documents live, you classify how automation can reach them. There are five realistic access paths, and they are not equal in effort, reliability, or cost.

  • Direct API access: Cleanest path. Credentials, rate limits, documented endpoints. Build time stays predictable.
  • Cloud storage with OAuth: Google Drive, SharePoint, Dropbox, Box. Usually solvable in a day once permissions are granted.
  • VPN-gated file share: Requires a persistent machine on the VPN. Adds infrastructure cost and a second point of failure.
  • Remote desktop / RPA: Used when the source system has no API. Fragile, slow, and expensive to maintain.
  • Manual export + upload: Not automation. It's a scheduled human task with a nicer wrapper.

If three of your document streams require remote desktop because the source software has no API, you are not scoping a document automation project. You are scoping an infrastructure project that happens to include automation. Price it that way.

2.5. The Security And Compliance Layer Most Teams Skip

Before the automation can touch a document, somebody has to approve the automation's right to touch it. At service businesses in the 11-100 employee range, this is usually the owner, an external MSP, or a compliance officer. All three are slow.

Run this checklist before you start building, not after:

  • Data residency: Where is the automation server located? Some contracts mandate in-country processing.
  • Credential storage: Are API keys and service account passwords stored in a secrets manager, not a workflow variable?
  • Audit logging: Can you produce a log of every document the automation touched in the last 90 days?
  • Access scope: Is the service account limited to the exact folder it needs, or does it have tenant-wide read access?
  • Termination procedure: If you fire the vendor tomorrow, can you revoke access in under 10 minutes?

If your vendor's answer to "where are the credentials stored" is a shrug or a screenshot of a workflow, walk away. That is the exact failure mode that turns a small automation into a breach notification.

3. Score The Project Before You Commit A Timeline

I use a simple scoring rubric on every scoping call. Ten questions, one point each. The score tells you whether the project is a 2-week build or a 2-month build.

QuestionYes = 1 Point
Does the source system have a documented API?+1
Are credentials already provisioned or provisioned within 3 days?+1
Can the automation run from a cloud server (not on-prem)?+1
Is the document format consistent (same template, same fields)?+1
Is there a single owner for the workflow on the client side?+1

Does the client have an IT contact who responds in under 48 hours?+1

Score interpretation:

  • 8-10 points: Standard build, 2-3 weeks, fixed-fee quote is safe.
  • 5-7 points: Discovery phase required. Quote the discovery separately, then the build.
  • 0-4 points: Do not fixed-fee this. Time and materials, or decline the scope.

I have turned down three projects this year that scored under 4. In all three cases, the client came back 3 months later after trying someone else and hitting exactly the wall the score predicted.

4. Separate The Discovery Phase From The Build Phase

The single biggest scoping improvement I've made in the last two years is splitting the engagement into two contracts. Discovery is one fixed-fee deliverable. Build is a second fixed-fee deliverable that's only quoted after discovery closes.

Discovery produces five artifacts:

  1. Document inventory: Every document type, location, volume, and owner.
  2. Access path map: How the automation will reach each document, with credentials verified.
  3. Security signoff: Written approval from whoever owns risk at the client.
  4. Test environment: A sandbox or staging account where the build can happen without production risk.
  5. Success metric: One sentence, one number, one deadline.

Typical discovery cost for a 11-100 employee service business runs $2,500 to $6,000 USD and takes 5 to 10 business days. That's cheap compared to the $15,000+ overruns that happen when you skip it.

If a vendor quotes the full build without doing discovery first, they are either pricing in a massive risk buffer or they are about to miss the deadline. Both are worse than a paid discovery phase.

5. Build For The Volume You Have, Not The Volume You Wish You Had

A recurring failure mode is over-engineering for volume that never materializes. A construction firm processing 40 submittals a week does not need a queue-based microservice architecture. They need a reliable n8n workflow running on a $20/month Hetzner box with Supabase for state.

Match the infrastructure to actual document volume:

  • Under 100 docs/day: Single n8n instance, SQLite or Supabase free tier, one webhook endpoint. Total infra: $20-40/month.
  • 100-500 docs/day: n8n with queue mode, Postgres backend, Redis for queues. Total infra: $60-120/month.
  • 500-2,000 docs/day: Horizontally scaled workers, dedicated Postgres, observability stack. Total infra: $200-500/month.
  • 2,000+ docs/day: Custom infrastructure conversation. n8n still works, but architecture decisions matter more than tool choice.

The mistake I see vendors make is quoting tier 3 infrastructure for a tier 1 problem, then billing ongoing management fees against infrastructure the client never needed. Scope to volume.

6. Make The Handoff Document Boring And Complete

The final deliverable is not the automation. It's the handoff document that lets the client's team run, audit, and extend the system without calling you every Tuesday. A good handoff document has six sections:

  • Architecture diagram: One page, every system and data flow labeled.
  • Credential inventory: What exists, where it's stored, who owns rotation.
  • Runbook: How to restart the workflow, where to find logs, what to do when a document fails to process.
  • Monitoring setup: What alerts fire, who gets paged, what the thresholds mean.
  • Change log template: How to document future modifications without breaking the audit trail.
  • Escalation contact: Who to call and under what conditions.

Clients who get this document stay clients for years. Clients who don't get this document call you at 11pm on a Sunday when the workflow breaks, then blame you for the fact that they can't fix it themselves.

7. Price For The Ongoing Relationship, Not The One-Time Build

Document automation projects have a tail. Source systems update their APIs, vendors change authentication methods, document templates get tweaked by someone in operations who didn't tell anyone. Budget for 8-15% of the build cost per year in maintenance, minimum.

Single document stream, API-accessible$4,000-8,000$400-8003-5 years

RPA-dependent workflows look cheaper upfront until you model the 5-year cost. Source systems break RPA constantly. An API integration with a 4-year lifespan and $600/year maintenance is almost always cheaper than an RPA build with a 1-year lifespan and $5,000/year maintenance, even if the RPA build quote is lower.

The cheapest automation at signing is rarely the cheapest automation at renewal. Model the 3-year total cost before picking an access path.
