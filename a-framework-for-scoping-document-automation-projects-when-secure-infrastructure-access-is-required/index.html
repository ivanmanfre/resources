This checklist fixes that.

What To Do With This

Before you start your next document automation project - or before you hire someone to build one for you - run through this checklist. Takes about 30 minutes if you have your IT person nearby. Could save you 2-4 weeks of delays and 40%+ in budget overruns.

The checklist works in three layers because that's how these projects actually break down:

1. The Document Layer - What are we actually automating?
2. The Access Layer - How does the automation get to the files?
3. The Action Layer - What happens after the automation runs?

Most teams nail Layer 1 and completely skip Layer 2. Then they're surprised when the project stalls.

Why This Matters

I was on a call with an insurance company a few months back. They needed digital signatures on policy documents. Simple enough, right? The workflow was clear: document gets approved in their system, someone needs to sign every page, then it goes back to the customer.

The actual automation? Straightforward. DocuSign or Adobe Acrobat Sign, connect it to their email trigger, done.

The problem? Their documents lived on a cloud server that required VPN access plus username and password. If you're in the office, just credentials. Off-site, you need VPN. And the automation doesn't sit in their office.

We ended up exploring three different approaches: RPA with browser simulation, remote machine access, API calls through their server. What should have been a 1-week build turned into a 3-week scoping conversation just to figure out how the automation would even touch the files.

This is the pattern. The automation logic is rarely the hard part. The infrastructure access is where projects die.

Another client, construction company, needed AI to read civil engineering plans and extract specific data. Cover page, vicinity map, site plan, erosion control plan - four drawings out of sometimes 300-page PDFs that crash computers when you try to open them.

The AI part? Fine. Gemini can read PDFs. The access part? They needed automated folders, clear file naming conventions, and a way to filter down massive documents before the AI even touched them. The scoping for that infrastructure layer took longer than building the actual extraction workflow.

Point is: if you don't scope the access layer properly, you're not scoping the project. You're guessing at the project.

The 3-Layer Checklist

Layer 1: The Document Layer

What are we actually automating?

Document Identification

  • [ ] What specific document types are in scope? (policy schedules, contracts, invoices, engineering drawings, etc.)
  • [ ] What format are they in? (PDF, Word, Crystal Reports export, etc.)
  • [ ] What's the typical file size? (Under 10MB is easy. 100MB+ needs different handling.)
  • [ ] How many documents per day/week/month?
  • [ ] Are there multiple versions or templates of the same document type?
Document Structure
  • [ ] Is the document structure consistent or does it vary?
  • [ ] How many pages typically need processing?
  • [ ] Are there specific pages that matter vs. pages that don't? (Like the construction client who only needed 4 drawings out of 300 pages)
  • [ ] Does the automation need to read the document, modify it, or both?
Signature Requirements (if applicable)
  • [ ] How many people need to sign?
  • [ ] Do they all sign the same pages or different pages?
  • [ ] Does every page need a signature or just specific ones?
  • [ ] Is a digital signature legally required or just a visual stamp?
  • [ ] What's your current signing tool? (Adobe Acrobat, DocuSign, wet signature scanned back in?)
TBH most projects stall here because nobody asks about file sizes and document structure upfront. A 10MB PDF and a 300MB PDF are completely different automation problems.

Layer 2: The Access Layer

How does the automation get to the files?

This is the layer that kills projects. Every question here that you can't answer is a potential 1-2 week delay.

Current Storage Location

  • [ ] Where do the documents live today? (Local server, cloud server, SharePoint, Google Drive, specific software system?)
  • [ ] Is it a cloud-based server or on-premise?
  • [ ] Do you need VPN to access it remotely?
  • [ ] What credentials are required? (username/password, SSO, MFA?)
Access Methods Available
  • [ ] Does the storage system have an API? (Check this. Seriously. Many do and nobody knows.)
  • [ ] Can files be accessed via direct URL or do they require navigation through a system?
  • [ ] Is there a way to export files programmatically from your source system?
  • [ ] Do you have remote machines or servers the automation could run on inside your network?
Security Constraints
  • [ ] What's your IT security policy on external tools accessing internal files?
  • [ ] Can automation tools be whitelisted for access?
  • [ ] Are there compliance requirements that restrict how files can be moved or processed?
  • [ ] Who needs to approve external system access? (Get their name. Loop them in now, not week three.)
Access Pattern
  • [ ] Will the automation need read-only access or read/write?
  • [ ] Does it need to move files between locations?
  • [ ] Does it need to save processed files back to the same location or a different one?
  • [ ] Can processed files overwrite originals or do they need new names? (e.g., "document_signed.pdf")
The question I always ask: "If I'm sitting at a coffee shop with a laptop and no VPN, can I access these files?" If the answer is no, we need to figure out how the automation will get access. That conversation needs to happen before anyone writes a single workflow.

Layer 3: The Action Layer

What happens after the automation runs?

Triggers

  • [ ] What kicks off the automation? (Email received, file dropped in folder, form submitted, manual button click?)
  • [ ] Is the trigger reliable and consistent?
  • [ ] Can you create a dedicated trigger location? (e.g., a specific folder or email address for automation)
  • [ ] How quickly does the automation need to respond? (Real-time, within minutes, daily batch?)
Processing Steps
  • [ ] What needs to happen to the document? (Read data, add signature, merge with other data, convert format?)
  • [ ] Does it need human review at any point or is it fully automated?
  • [ ] What's the approval flow if human review is required?
  • [ ] Are there multiple people who might need to take action?
Output Destination
  • [ ] Where does the finished document go? (Back to original folder, emailed to customer, uploaded to another system?)
  • [ ] Does it need to be saved in multiple locations?
  • [ ] Is there a naming convention for processed files?
  • [ ] Do you need a record of what was processed and when?
Error Handling
  • [ ] What happens if a document can't be processed? (Wrong format, missing data, system unavailable?)
  • [ ] Who gets notified when something fails?
  • [ ] Is there a manual fallback process?

Implementation Priority

If you're short on time, these are the three questions that predict 80% of project delays:

1. Does the storage system have an API or will we need VPN/remote access? This single question determines whether your project is a 1-week build or a 4-week infrastructure conversation.

2. How big are the files and how many per day? Small files in low volume = simple. Large files in high volume = need to rethink the architecture.

3. Who approves external system access and have they been looped in? The number of projects I've seen stall because IT security wasn't consulted until week three is genuinely depressing.

Rough Timeline Estimates

Based on what I've seen across projects:

    • Access Scenario: API available, no VPN required | Added Time to Project: +0 days
    • Access Scenario: API available, VPN required | Added Time to Project: +1-2 weeks (VPN setup, testing)
    • Access Scenario: No API, need RPA/browser automation | Added Time to Project: +2-3 weeks (build + stability testing)
    • Access Scenario: No API, need remote machine setup | Added Time to Project: +2-4 weeks (IT coordination, setup, testing)
    • Access Scenario: Security approval required | Added Time to Project: +1-3 weeks (depends on your org)
These aren't exact, but they're close enough to stop you from quoting a 2-week project that's actually a 6-week project.

Putting It All Together

Before you start any document automation project, run through this checklist with your team. If you're hiring someone to build it, send them this checklist filled out. You'll look like you know what you're doing (because you will), and you'll avoid the back-and-forth that adds weeks to timelines.

The projects that go smoothly are the ones where Layer 2 is figured out before anyone opens an automation tool. The projects that blow up are the ones where someone says "we'll figure out access later."

There is no later. There's only "the project stalled at week two."

Your Next 7 Days

Day 1-2: Fill out Layer 1 for your highest-priority document automation project. Get specific about document types, volumes, and file sizes.

Day 3-4: Work through Layer 2 with your IT person. Find out if APIs exist, what access methods are possible, and who needs to approve external access.

Day 5: Complete Layer 3. Map the triggers, processing steps, and destinations.

Day 6-7: Review the full checklist. Any question you couldn't answer is a potential delay. Get those answers before you start building or hiring.
